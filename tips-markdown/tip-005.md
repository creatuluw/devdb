# DevTip #005 — Migrations as a Safety Harness for Agents

**Category:** Data Architecture

**Subtitle:** Make schema changes legible, reversible, and reviewable

---

## Why this matters

Agents can generate SQL fast, but they are also excellent at accidentally destroying invariants: adding nullable columns without backfills, rewriting tables without indexes, or shipping migrations that work locally but break in CI. A principled migration workflow turns database changes into a contract the agent can follow and you can approve in minutes.


## 1. Use a 3-file migration pattern (up / down / verify)

Write migrations as an explicit plan with a validation step. The key is to separate *DDL* from *data backfills* and to include a cheap verification query that can run in CI after applying the migration.


```sql
-- 2026-04-11_001_add_user_slug.up.sql
ALTER TABLE users
  ADD COLUMN slug text;

CREATE UNIQUE INDEX CONCURRENTLY IF NOT EXISTS users_slug_uq
  ON users (slug)
  WHERE slug IS NOT NULL;
```


```sql
-- 2026-04-11_001_add_user_slug.verify.sql
-- Expect: 0 rows
SELECT id
FROM users
WHERE slug IS NULL
  AND created_at < now() - interval '10 minutes';
```


## 2. Make backfills idempotent (so retries are safe)

Agents (and CI) will retry. Your backfill should be safe to run multiple times and should converge. Prefer chunked updates with a stable ordering to avoid long locks.


```sql
-- 2026-04-11_002_backfill_user_slug.up.sql
-- Repeat-safe backfill: only fills missing slugs.
WITH batch AS (
  SELECT id
  FROM users
  WHERE slug IS NULL
  ORDER BY id
  LIMIT 500
)
UPDATE users u
SET slug = lower(regexp_replace(u.name, '[^a-zA-Z0-9]+', '-', 'g')) || '-' || u.id
FROM batch
WHERE u.id = batch.id;
```


## 3. Encode invariants in the database (not in prompts)

If an invariant matters, enforce it with constraints. Prompts are not enforcement mechanisms. This is especially important for agent-written CRUD because agents tend to miss edge cases.


```sql
-- 2026-04-11_003_enforce_slug_not_null.up.sql
-- Only after verify.sql passes for a few deploys:
ALTER TABLE users
  ALTER COLUMN slug SET NOT NULL;
```


## 4. Teach the agent your migration & review checklist

Put a short checklist in your repo (for example: docs/AGENTS.md) and refer to it in your system prompt. The agent should output: affected tables, lock risk, rollback plan, and verification queries.


```md
# docs/AGENTS.md (excerpt)

## DB Change Checklist
- [ ] Migration includes verify.sql with an assertion query
- [ ] Backfill is idempotent and chunked (LIMIT)
- [ ] New constraints shipped only after data is clean
- [ ] Indexes use CONCURRENTLY when safe
- [ ] Rollback path is documented (down.sql or manual note)
```


---

## Pro Move

**Add a migration CI gate with a disposable Postgres**

In CI, spin up Postgres, apply all migrations from scratch, then run every verify.sql. If the agent proposes a migration that cannot be applied cleanly in a fresh database, you catch it before prod. This also forces deterministic migrations: no environment-specific assumptions, no manual steps, no hidden ordering bugs.
