# DevTip #002 — Agent-First API Design
### Types Are the Spec

> **Why this matters:** A well-typed, self-documenting TypeScript API surface lets an LLM agent reason about, extend, and debug your routes without ever needing you to explain business logic — your code *is* the documentation.

---

## The Agent's Mental Model of Your API

When an agent reads a SvelteKit `+server.ts` route, it constructs an internal model of: what comes in, what goes out, what can fail, and what the business intent is. The more ambiguity it encounters — untyped request bodies, `any` returns, unnamed magic strings — the more it will fill gaps with plausible-but-wrong assumptions.

Your job is to make that mental model construction **deterministic**.

---

## Consistent Naming Across the Boundary

Pick a naming convention and codify it everywhere. A clean pattern:

```
GET    /api/invoices           → listInvoices
GET    /api/invoices/:id       → getInvoice
POST   /api/invoices           → createInvoice
PATCH  /api/invoices/:id       → updateInvoice
DELETE /api/invoices/:id       → deleteInvoice
```

Reflect this naming in your service module:

```typescript
// features/invoices/invoiceService.ts

export async function listInvoices(filters: InvoiceFilters): Promise<Invoice[]> { ... }
export async function getInvoice(id: string): Promise<Invoice | null> { ... }
export async function createInvoice(input: CreateInvoiceInput): Promise<Invoice> { ... }
export async function updateInvoice(id: string, patch: UpdateInvoiceInput): Promise<Invoice> { ... }
export async function deleteInvoice(id: string): Promise<void> { ... }
```

When an agent reads the route and the service, the one-to-one correspondence is instantly obvious.

## Zod Schemas as the Single Source of Truth

```typescript
// features/invoices/invoice.schemas.ts
import { z } from 'zod';

export const CreateInvoiceSchema = z.object({
  clientId: z.string().uuid(),
  lineItems: z.array(
    z.object({
      description: z.string().min(1).max(500),
      quantity: z.number().int().positive(),
      unitPrice: z.number().int().positive(), // cents
    })
  ).min(1),
  dueAt: z.string().datetime(), // ISO 8601
});

export type CreateInvoiceInput = z.infer<typeof CreateInvoiceSchema>;
```

Your route becomes trivially readable:

```typescript
// routes/api/invoices/+server.ts
export const POST: RequestHandler = async ({ request }) => {
  const body = await request.json();
  const parsed = CreateInvoiceSchema.safeParse(body);

  if (!parsed.success) {
    throw error(400, { message: 'Invalid input', issues: parsed.error.issues });
  }

  const invoice = await createInvoice(parsed.data);
  return json(invoice, { status: 201 });
};
```

An agent handed this file plus `invoice.schemas.ts` can immediately: validate correctness, add new fields, write tests, or generate a client SDK — all without asking you anything.

## JSDoc as Agent-Readable Spec

JSDoc isn't just for IDE tooltips. It's structured natural language that agents parse as specification:

```typescript
/**
 * Creates a new invoice in DRAFT status.
 *
 * - Calculates `totalCents` from lineItems automatically.
 * - Sends no notifications — use `sendInvoice()` to transition to SENT.
 * - Requires the `clientId` to reference an existing Client record.
 *
 * @throws {ClientNotFoundError} if the clientId does not exist
 * @throws {DatabaseError} on persistence failure
 */
export async function createInvoice(input: CreateInvoiceInput): Promise<Invoice> {
```

This comment answers the questions an agent would otherwise hallucinate: *Does this send an email? Does it calculate totals? What errors can it throw?*

## Typed Error Envelopes

```typescript
// lib/api/response.ts

export interface ApiError {
  code: string;       // machine-readable: 'INVOICE_NOT_FOUND'
  message: string;    // human-readable
  issues?: unknown;   // Zod validation issues, if applicable
}

export interface ApiSuccess<T> {
  data: T;
  meta?: {
    total?: number;
    page?: number;
  };
}
```

When every route returns `ApiSuccess<T>` or `ApiError`, the agent can write client-side error handling correctly on the first attempt.

## Pro Move: Generate OpenAPI Spec from Zod at Build Time

Use [`zod-to-openapi`](https://github.com/asteasolutions/zod-to-openapi) to auto-generate an OpenAPI spec from your schemas. Feed this spec to your agent as a compact, structured description of your entire API surface — far more token-efficient than pasting individual route files.

```typescript
// scripts/generate-openapi.ts
import { generateOpenApi } from '@anatine/zod-nestjs';
// ... register your schemas, output to openapi.json
```

Then your agent prompt becomes: *"Here is the OpenAPI spec for this project. Add a new endpoint for invoice PDF export following the existing patterns."* One context file covers your whole API.
