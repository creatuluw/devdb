# DevTip #004 — Tools aren’t “APIs”; they’re contracts: design tool schemas like you design libraries

**Subtitle:** If a tool can’t fail predictably, your agent can’t recover.

**Category:** Agent Tooling

## Why this matters
Most agent failures aren’t “model problems” — they’re interface problems. If your tool call surface is underspecified (ambiguous inputs, untyped outputs, inconsistent errors), the agent can’t plan, can’t validate, and can’t recover. A well-designed tool schema gives the model a *deterministic affordance*: it knows what it can ask for, what it will get back, and how to react when something goes wrong.

If you treat tools as first-class libraries (with types, invariants, and error taxonomies), you get:
- fewer invalid tool calls
- cheaper prompts (less “explaining”)
- better multi-step planning
- reliable retries and fallbacks

---

## The core idea: make tools machine-legible and failure-aware

### 1) Use discriminated unions for results
Return a single top-level shape that is always valid JSON and always parseable.

```ts
// shared/types/tool-result.ts
export type ToolOk<T> = {
  ok: true;
  data: T;
  meta?: {
    requestId?: string;
    durationMs?: number;
  };
};

export type ToolErr = {
  ok: false;
  error: {
    code:
      | "INVALID_INPUT"
      | "NOT_FOUND"
      | "UNAUTHORIZED"
      | "RATE_LIMITED"
      | "CONFLICT"
      | "UPSTREAM_FAILURE"
      | "TIMEOUT"
      | "INTERNAL";
    message: string;
    retryable: boolean;
    details?: unknown;
  };
  meta?: {
    requestId?: string;
    durationMs?: number;
  };
};

export type ToolResult<T> = ToolOk<T> | ToolErr;
```

Why: the agent can branch immediately on `ok` without brittle string matching.

### 2) Validate inputs with Zod and echo normalized inputs
Agents often send nearly-correct inputs. Normalize them and return what you actually used.

```ts
import { z } from "zod";

export const SearchTicketsInput = z.object({
  query: z.string().min(1),
  limit: z.number().int().min(1).max(50).default(20),
  cursor: z.string().optional(),
});

export type SearchTicketsArgs = z.infer<typeof SearchTicketsInput>;
```

```ts
export type SearchTicketsData = {
  normalized: SearchTicketsArgs;
  items: Array<{ id: string; title: string; status: "open" | "closed" }>;
  nextCursor?: string;
};
```

Why: when a later step uses the tool output as context, the agent sees the exact canonical args.

### 3) Design error codes for *agent decisions*
Humans like messages; agents need categories.

Good error codes are:
- **actionable** (imply a next step)
- **stable** (don’t change across refactors)
- **complete** (cover expected failures)

Example mapping:
- `INVALID_INPUT` → fix args (no retry)
- `RATE_LIMITED` → retry with backoff
- `UPSTREAM_FAILURE` → switch provider / degrade gracefully
- `CONFLICT` → re-read state then retry

### 4) Make retries safe: add idempotency and optimistic concurrency
If the agent retries, you must guarantee the operation is safe.

```ts
export const CreateInvoiceInput = z.object({
  idempotencyKey: z.string().min(10),
  customerId: z.string(),
  amountCents: z.number().int().positive(),
});
```

For updates:

```ts
export const UpdateDocInput = z.object({
  docId: z.string(),
  ifVersion: z.number().int().nonnegative(),
  patch: z.array(z.object({ op: z.enum(["replace"]), path: z.string(), value: z.any() })),
});
```

Why: agents are inherently retry-prone (timeouts, tool glitches, partial context). Concurrency guards prevent “double writes”.

### 5) Add a lightweight “capabilities” tool
Agents do better when they can ask what the tool can do.

```ts
export type Capabilities = {
  name: string;
  version: string;
  operations: Array<{
    op: string;
    description: string;
    inputSchemaRef: string;
    outputSchemaRef: string;
    commonErrors: string[];
  }>;
};
```

Why: it reduces prompt bloat and helps the agent select the right operation.

## Pro Move: bake a tool-contract test suite into CI
Add “contract tests” that verify:
- every tool returns `ToolResult<T>`
- Zod validation runs and errors map to `INVALID_INPUT`
- retryable errors set `retryable: true`
- outputs remain backward compatible (snapshot the JSON schema)

A minimal example with Vitest:

```ts
import { describe, expect, it } from "vitest";
import { searchTickets } from "../tools/searchTickets";

describe("tool contract: searchTickets", () => {
  it("always returns ToolResult shape", async () => {
    const res = await searchTickets({ query: "" } as any);
    expect(res).toHaveProperty("ok");
    if (res.ok === false) {
      expect(res.error).toHaveProperty("code");
      expect(typeof res.error.retryable).toBe("boolean");
    }
  });
});
```

If you treat tool schemas as public APIs and enforce them with tests, agents become predictably capable — and failures become recoverable rather than catastrophic.
