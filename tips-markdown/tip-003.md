# DevTip #003 — Prompt Chaining for UI Generation
### Consistent Components Every Time

> **Why this matters:** One-shot UI generation from LLM agents produces inconsistent, design-system-ignorant components — breaking the chain into discrete, primed steps produces output you can ship without a rewrite.

---

## Why One-Shot UI Generation Fails

Ask an agent to "build a SvelteKit invoice status badge with Tailwind" and you'll get something that works but:

- Uses arbitrary color classes instead of your design tokens
- Ignores your component conventions (slot patterns, prop naming)
- Has accessibility gaps
- Won't match any other component in your system

The agent wasn't wrong — it was under-informed. Prompt chaining fixes this by feeding the right context at each stage of reasoning, before any code is written.

---

## The 4-Step UI Generation Chain

### Step 1 — Prime with Design System Context

Before asking for any code, inject your design system as a structured document:

```typescript
// lib/design/tokens.ts

export const colors = {
  status: {
    draft:   { bg: 'bg-zinc-100',   text: 'text-zinc-600',   border: 'border-zinc-200'   },
    sent:    { bg: 'bg-blue-50',    text: 'text-blue-700',   border: 'border-blue-200'   },
    paid:    { bg: 'bg-green-50',   text: 'text-green-700',  border: 'border-green-200'  },
    overdue: { bg: 'bg-red-50',     text: 'text-red-700',    border: 'border-red-200'    },
  },
} as const;
```

**Prompt 1 (priming):**
> "Here is our design token file: [paste tokens.ts]. Here is an example of an existing badge component we use for user roles: [paste RoleBadge.svelte]. Do not write any code yet. Confirm you understand our color token structure and component conventions."

Let the agent confirm and summarize. This forces it to internalize your system before generating.

---

### Step 2 — Specify the Contract

**Prompt 2 (contract):**
> "We need an `InvoiceStatusBadge` component. Here is the TypeScript type it must accept:
>
> ```typescript
> import type { InvoiceStatus } from '$features/invoices/invoice.types';
>
> interface Props {
>   status: InvoiceStatus;
>   size?: 'sm' | 'md';      // default: 'md'
>   showIcon?: boolean;       // default: true
> }
> ```
>
> Do not write the component yet — describe the mapping of each status to its visual treatment and which icon you'd use for each state."

This step externalizes the agent's reasoning. You can catch wrong assumptions before a single line of Svelte is written.

---

### Step 3 — Generate the Component

**Prompt 3 (generation):**
> "Now write the `InvoiceStatusBadge.svelte` component using:
> - The contract from Step 2
> - Our design tokens (not arbitrary Tailwind classes)
> - The same prop/slot pattern as `RoleBadge.svelte`
> - Tailwind v4 syntax (no `@apply`, use class utilities directly)
> - Full TypeScript, no `any`
> - ARIA role='status' and a visually hidden label for screen readers"

```svelte
<script lang="ts">
  import type { InvoiceStatus } from '$features/invoices/invoice.types';
  import { colors } from '$lib/design/tokens';

  interface Props {
    status: InvoiceStatus;
    size?: 'sm' | 'md';
    showIcon?: boolean;
  }

  let { status, size = 'md', showIcon = true }: Props = $props();

  const token = colors.status[status];
  const sizeClass = size === 'sm' ? 'text-xs px-2 py-0.5' : 'text-sm px-2.5 py-1';
</script>

<span
  role="status"
  class="inline-flex items-center gap-1.5 rounded-full border font-medium {token.bg} {token.text} {token.border} {sizeClass}"
>
  {#if showIcon}
    <span aria-hidden="true"><!-- icon per status --></span>
  {/if}
  <span class="sr-only">Invoice status:</span>
  {status}
</span>
```

---

### Step 4 — Harden and Extend

**Prompt 4 (hardening):**
> "Now add:
> 1. A Storybook story with all 4 status variants shown at both sizes
> 2. A `data-testid` attribute using the pattern `status-badge-{status}`
> 3. Export this component from `features/invoices/index.ts`"

Keeping this as a separate step prevents the generation step from getting bloated with non-component concerns.

---

## One-Shot vs. Multi-Turn: When to Use Each

| Scenario | Use |
|---|---|
| New component following a clear existing pattern | One-shot with a rich example + token context |
| Novel component type (no existing pattern) | Multi-turn chain (all 4 steps) |
| Modifying an existing component | One-shot with the component file + diff instructions |
| Full feature UI (multiple components) | Multi-turn: design → contract → generate each → integrate |
| Rapid prototype / throwaway | One-shot, accept imperfection |

---

## Pro Move: Snapshot Your Design Context as a Reusable Prompt Prefix

Maintain a `agent-ui-context.md` file that you paste at the start of every UI generation session. Include:

1. Your Tailwind config (color palette, spacing scale, font sizes)
2. Two "exemplar" components — one simple, one complex — that represent your ideal output
3. Your component file naming and prop naming conventions
4. Any banned patterns (e.g., "never use inline styles," "never use `!important`," "always use `$props()` rune, not `export let`")

With a well-maintained prefix, Step 1 becomes a paste, not a task — and your agent starts every UI session already inside your design system.
