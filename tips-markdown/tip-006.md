# DevTip #006 — Tokens as a Design Budget

**Subtitle:** Turn prompt length into an explicit cost model for features

**Category:** Design Systems

## Why this matters
Most teams treat prompt size as an accident: context grows until latency spikes, bills jump, and the agent starts missing key details. Treat tokens like CSS utility classes: a constrained palette that forces consistency. If you make **token usage a first-class budget**, you get three compounding wins:

1. **Predictable cost/latency** per feature.
2. **More legible prompts** (agents succeed more often when the structure is tight).
3. **A real design system for language**: reusable, composable prompt parts.

## The core idea: a token budget table
Define a budget by *prompt layers* and enforce it in code. A practical default for a “feature-level” agent call:

- System instructions: 10–15%
- Project / app context: 15–25%
- Retrieved knowledge (RAG): 20–40%
- User request: 5–10%
- Output allowance: 20–30%

This turns “we ran out of context” into “RAG exceeded its allocation.”

### Example: encode budgets as config

```ts
// llm/budgets.ts
export type BudgetKey =
  | "system"
  | "appContext"
  | "retrieval"
  | "user"
  | "output";

export const DEFAULT_BUDGET: Record<BudgetKey, number> = {
  system: 700,
  appContext: 900,
  retrieval: 1600,
  user: 350,
  output: 900,
};

export const TOTAL_BUDGET = Object.values(DEFAULT_BUDGET).reduce(
  (a, b) => a + b,
  0
);
```

### Example: trim each layer deterministically

```ts
// llm/trim.ts
import { encoding_for_model } from "tiktoken";

export function truncateToTokens(
  input: string,
  maxTokens: number,
  model: string = "gpt-4o-mini"
) {
  const enc = encoding_for_model(model);
  const tokens = enc.encode(input);

  if (tokens.length <= maxTokens) return input;

  // Keep the *end* for logs / recent context, but you can invert this.
  const slice = tokens.slice(tokens.length - maxTokens);
  const out = enc.decode(slice);
  enc.free();
  return out;
}
```

### Example: build prompts from a “prompt design system”
Co-locate prompt “components” like you would UI components.

```ts
// llm/promptParts.ts
export function systemPreamble() {
  return [
    "You are a senior TypeScript engineer.",
    "Return valid JSON that matches the schema.",
    "If unsure, ask for clarification in an 'issues' array.",
  ].join("\n");
}

export function brandVoice() {
  return [
    "Style: concise, direct, no fluff.",
    "Prefer bullet lists over paragraphs.",
  ].join("\n");
}
```

Then assemble with budgets:

```ts
import { DEFAULT_BUDGET } from "./budgets";
import { truncateToTokens } from "./trim";
import { systemPreamble, brandVoice } from "./promptParts";

export function buildPrompt({ appContext, retrieved, user }: {
  appContext: string;
  retrieved: string;
  user: string;
}) {
  const system = truncateToTokens(
    systemPreamble() + "\n\n" + brandVoice(),
    DEFAULT_BUDGET.system
  );

  const ctx = truncateToTokens(appContext, DEFAULT_BUDGET.appContext);
  const rag = truncateToTokens(retrieved, DEFAULT_BUDGET.retrieval);
  const usr = truncateToTokens(user, DEFAULT_BUDGET.user);

  return {
    system,
    ctx,
    rag,
    usr,
  };
}
```

## Practical enforcement: fail closed when budgets are violated
If you only truncate, you may silently remove crucial facts. Add an explicit “budget report” and fail closed for critical tasks.

```ts
export function budgetReport(parts: Record<string, string>) {
  // If you can, compute real token counts; string length is a quick proxy.
  return Object.fromEntries(
    Object.entries(parts).map(([k, v]) => [k, v.length])
  );
}

export function assertNoHardTruncation(
  original: string,
  truncated: string,
  label: string
) {
  if (original !== truncated) {
    throw new Error(
      `Budget exceeded for ${label}. Refuse to run with missing context.`
    );
  }
}
```

Use “refuse to run” for high-stakes operations (billing, deletes, permissions) and “truncate + warn” for low-stakes.

## Pro Move
**Add token budgets to code review like you add design tokens to UI review.** Create a PR checklist item:

- “Did we change prompt parts?”
- “Did we update the budget table?”
- “Did we add/adjust a budgeted retrieval query?”

Then wire a CI check that prints the budget report for key agent flows. You will catch prompt bloat *before* it hits production latency and cost.
