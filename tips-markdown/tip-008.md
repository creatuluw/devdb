# DevTip #008 -- Contract Tests for Agent Tool Interfaces

**Category:** Testing & Quality

## Why This Matters

Agent-written code ships fast -- so fast that the seams between your tools and the agent that calls them become the highest-risk surface in your stack. A model will confidently call a tool with the wrong argument shape, expect a field that got renamed, or retry on an error it should abort on. Contract tests lock the tool interface down. If the contract breaks, CI breaks -- before the agent ever runs.

## 1. The ToolResult contract

Every tool should return a discriminated union. The contract test verifies the shape, not the data:

```typescript
// lib/tool-result.ts
export type ToolOk<T> = { ok: true; data: T };
export type ToolErr = {
  ok: false;
  error: { code: string; message: string; retryable: boolean };
};
export type ToolResult<T> = ToolOk<T> | ToolErr;
```

Contract test:

```typescript
// tests/contracts/tool-shape.test.ts
import { describe, expect, it } from "vitest";
import { searchDocs } from "$lib/tools/searchDocs";

describe("contract: searchDocs", () => {
  it("always returns ToolResult shape", async () => {
    const result = await searchDocs({ query: "" } as any);
    expect(result).toHaveProperty("ok");
  });

  it("maps validation failure to INVALID_INPUT", async () => {
    const result = await searchDocs({ query: "" } as any);
    if (!result.ok) {
      expect(result.error.code).toBe("INVALID_INPUT");
      expect(typeof result.error.retryable).toBe("boolean");
    }
  });

  it("retryable is false for INVALID_INPUT", async () => {
    const result = await searchDocs({ query: "" } as any);
    if (!result.ok && result.error.code === "INVALID_INPUT") {
      expect(result.error.retryable).toBe(false);
    }
  });
});
```

## 2. Zod schema as the contract source of truth

Co-locate the Zod schema with the tool. The schema IS the contract -- derive types from it, never the other way:

```typescript
// lib/tools/searchDocs.ts
import { z } from "zod";
import type { ToolResult } from "$lib/tool-result";

export const SearchDocsInput = z.object({
  query: z.string().min(1),
  limit: z.number().int().min(1).max(50).default(10),
});

export type SearchDocsArgs = z.infer<typeof SearchDocsInput>;

export type SearchDocsData = {
  items: Array<{ id: string; title: string; excerpt: string }>;
  total: number;
};

export async function searchDocs(
  raw: unknown
): Promise<ToolResult<SearchDocsData>> {
  const parsed = SearchDocsInput.safeParse(raw);
  if (!parsed.success) {
    return {
      ok: false,
      error: {
        code: "INVALID_INPUT",
        message: parsed.error.message,
        retryable: false,
      },
    };
  }
  // ... implementation
}
```

## 3. Output schema contract test

Test that the output shape never silently regresses:

```typescript
import { z } from "zod";

const SearchDocsOutputSchema = z.object({
  ok: z.literal(true),
  data: z.object({
    items: z.array(z.object({
      id: z.string(),
      title: z.string(),
      excerpt: z.string(),
    })),
    total: z.number(),
  }),
});

it("success output matches schema", async () => {
  const result = await searchDocs({ query: "sveltekit", limit: 3 });
  if (result.ok) {
    expect(() => SearchDocsOutputSchema.parse(result)).not.toThrow();
  }
});
```

## Pro Move

Add a `tools-contract.test.ts` barrel that imports and runs contract tests for every tool in your toolbox. Wire it as a separate Vitest project in `vitest.config.ts` so it runs in CI independently from unit tests -- and fails loudly before any agent session starts. Think of it as a pre-flight check: if any tool contract is broken, the agent never launches.
