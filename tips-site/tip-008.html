DEVTIP #008 · Testing & Quality

Contract Tests for Agent Tool Interfaces

Every tool your agent calls needs a test that runs in CI

fact_check

WHY THIS MATTERS

"Agent-written code ships fast — so fast that the seams between your tools and the agent that calls them become the highest-risk surface in your stack. A model will confidently call a tool with the wrong argument shape, expect a field that got renamed, or retry on an error it should abort on. Contract tests lock the interface down. If the contract breaks, CI breaks — before the agent ever runs."

1. The ToolResult contract

Every tool returns a discriminated union. The contract test verifies the shape, not the data:

// lib/tool-result.ts
export type ToolOk<T> = { ok: true; data: T };
export type ToolErr = {
  ok: false;
  error: { code: string; message: string; retryable: boolean };
};
export type ToolResult<T> = ToolOk<T> | ToolErr;

2. Contract test in Vitest

Three assertions cover the full contract: shape, error code, and retryable flag:

// tests/contracts/tool-shape.test.ts
import { describe, expect, it } from "vitest";
import { searchDocs } from "$lib/tools/searchDocs";

describe("contract: searchDocs", () => {
  it("always returns ToolResult shape", async () => {
    const result = await searchDocs({ query: "" } as any);
    expect(result).toHaveProperty("ok");
  });

  it("maps validation failure to INVALID_INPUT", async () => {
    const result = await searchDocs({ query: "" } as any);
    if (!result.ok) {
      expect(result.error.code).toBe("INVALID_INPUT");
      expect(typeof result.error.retryable).toBe("boolean");
    }
  });

  it("retryable is false for INVALID_INPUT", async () => {
    const result = await searchDocs({ query: "" } as any);
    if (!result.ok && result.error.code === "INVALID_INPUT") {
      expect(result.error.retryable).toBe(false);
    }
  });
});

3. Zod schema as contract source of truth

Co-locate the schema with the tool. The schema IS the contract — derive types from it, never the other way:

// lib/tools/searchDocs.ts
export const SearchDocsInput = z.object({
  query: z.string().min(1),
  limit: z.number().int().min(1).max(50).default(10),
});

export type SearchDocsArgs = z.infer<typeof SearchDocsInput>;

export async function searchDocs(
  raw: unknown
): Promise<ToolResult<SearchDocsData>> {
  const parsed = SearchDocsInput.safeParse(raw);
  if (!parsed.success) {
    return { ok: false, error: {
      code: "INVALID_INPUT",
      message: parsed.error.message,
      retryable: false,
    }};
  }
  // ... implementation
}

4. Output schema contract test

Test that the output shape never silently regresses:

const OutputSchema = z.object({
  ok: z.literal(true),
  data: z.object({
    items: z.array(z.object({
      id: z.string(),
      title: z.string(),
      excerpt: z.string(),
    })),
    total: z.number(),
  }),
});

it("success output matches schema", async () => {
  const result = await searchDocs({ query: "sveltekit", limit: 3 });
  if (result.ok) {
    expect(() => OutputSchema.parse(result)).not.toThrow();
  }
});

Pro Move

Add a contract barrel to CI as a pre-flight check

Create a tools-contract.test.ts barrel that imports and runs contract tests for every tool in your toolbox. Wire it as a separate Vitest project in vitest.config.ts so it runs in CI independently from unit tests.

Think of it as a pre-flight check: if any tool contract is broken, the agent never launches. This is the fastest way to catch interface regressions before they reach a running agent session.
