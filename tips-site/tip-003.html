DEVTIP #003

Prompt Chaining for UI Generation

Consistent Components Every Time

widgets

WHY THIS MATTERS

"One-shot UI generation from LLM agents produces inconsistent, design-system-ignorant components — breaking the chain into discrete, primed steps produces output you can ship without a rewrite."

Why One-Shot UI Generation Fails

Ask an agent to "build a SvelteKit invoice status badge with Tailwind" and you'll get something that works but:

  • close Uses arbitrary color classes instead of your design tokens
  • close Ignores your component conventions (slot patterns, prop naming)
  • close Has accessibility gaps
  • close Won't match any other component in your system

The agent wasn't wrong — it was under-informed. Prompt chaining fixes this by feeding the right context at each stage of reasoning, before any code is written.

The 4-Step UI Generation Chain

4 Steps
PRIME
SPEC
GEN
HARDEN
STEP 01

Prime with Design System Context

Before asking for any code, inject your design system as a structured document. Keep a DESIGN_SYSTEM.md in your project. 

// lib/design/tokens.ts

export const colors = {
  status: {
    draft:   { bg: 'bg-zinc-100',
               text: 'text-zinc-600' },
    sent:    { bg: 'bg-blue-50',
               text: 'text-blue-700' },
    paid:    { bg: 'bg-green-50',
               text: 'text-green-700' },
    overdue: { bg: 'bg-red-50',
               text: 'text-red-700' },
  },
} as const;
STEP 02

Specify the Contract

Define the component's interface explicitly before generation. This externalizes the agent's reasoning so you can catch wrong assumptions. 

import type { InvoiceStatus }
  from '$features/invoices/invoice.types';

interface Props {
  status: InvoiceStatus;
  size?: 'sm' | 'md';      // default: 'md'
  showIcon?: boolean;  // default: true
}
STEP 03

Generate the Component

Now, and only now, ask for the code — using the contract, your design tokens, and the same prop/slot pattern as existing components. 

<!-- InvoiceStatusBadge.svelte -->
<script lang="ts">
  let { status, size = 'md',
        showIcon = true }: Props
    = $props();

  const token = colors.status[status];
</script>

<span role="status"
  class="inline-flex items-center
    gap-1.5 rounded-full border
    font-medium {token.bg} {token.text}"
>
  {status}
</span>
STEP 04

Harden and Extend

Use a focused follow-up to close gaps rather than regenerating. Keep this as a separate step to prevent bloat.

1. Add a Storybook story with all 4 status variants at both sizes
2. Add data-testid using the pattern status-badge-{status}
3. Export from features/invoices/index.ts

One-Shot vs. Multi-Turn: When to Use Each

Scenario Use
New component following a clear existing pattern One-shot with a rich example + token context
Novel component type (no existing pattern) Multi-turn chain (all 4 steps)
Modifying an existing component One-shot with the component file + diff instructions
Full feature UI (multiple components) Multi-turn: design → contract → generate → integrate
Rapid prototype / throwaway One-shot, accept imperfection

Pro Move

Snapshot Your Design Context as a Reusable Prompt Prefix

Maintain an agent-ui-context.md file that you paste at the start of every UI generation session. Include:

01

Tailwind Config

Your color palette, spacing scale, and font sizes

02

Two Exemplar Components

One simple, one complex — representing your ideal output

03

Naming Conventions

Component file naming and prop naming patterns

04

Banned Patterns

"Never use inline styles," "never use !important," "always use $props() rune"

With a well-maintained prefix, Step 1 becomes a paste, not a task — and your agent starts every UI session already inside your design system.
