DEVTIP #007

Zero Downtime agent deployments on Railway

Ship without interrupting running agent sessions

DevOps and Infra
rocket_launch
WHY THIS MATTERS

Agents are often long running processes: they hold websocket sessions, stream tool calls, and keep in flight work. A naive redeploy kills the process and you lose the current step, the local cache, and any pending writes. Railway can do near zero downtime deploys, but only if you give it explicit readiness and shutdown signals.

The pattern: readiness plus draining

Build a release contract: the new instance must prove it is ready before it receives traffic, and the old instance must drain active sessions before it exits.

Health endpoint

Return 503 while draining so traffic stays on the old instance.

// src/routes/health/+server.ts
import { json } from '@sveltejs/kit';

let draining = false;

export function startDrain() {
  draining = true;
}

export function GET() {
  // 503 tells Railway to keep traffic on the old instance
  if (draining) return new Response('draining', { status: 503 });
  return json({ ok: true });
}

Railway deploy config

Tell the platform what readiness means.

# railway.toml
[deploy]
healthcheckPath = "/health"
healthcheckTimeout = 300
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 3

Drain on SIGTERM

Flip readiness first, then wait for active work to finish.

// src/hooks.server.ts
import { startDrain } from './routes/health/+server';

// Optional: keep a counter of active requests or sessions in your app.
let active = 0;

export async function handle({ event, resolve }) {
  active++;
  try {
    return await resolve(event);
  } finally {
    active--;
  }
}

process.on('SIGTERM', async () => {
  // 1) Flip readiness so Railway stops routing traffic here.
  startDrain();

  // 2) Wait for in flight work to finish.
  const deadlineMs = Date.now() + 20_000;
  while (active > 0 && Date.now() < deadlineMs) {
    await new Promise(r => setTimeout(r, 250));
  }

  // 3) Exit so the platform can replace the instance.
  process.exit(0);
});
PRO MOVE

Drain websockets, not just HTTP

Add a draining gate that is shared by both HTTP and websocket paths. If your agent streams tokens over websockets, track active socket count and only exit after sockets close or a deadline expires. Also record a deploy build id in logs so you can correlate errors to releases in minutes.
