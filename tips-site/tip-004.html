DEVTIP #004 · Agent Tooling

Tools aren't "APIs"; they're contracts

Design tool schemas like you design libraries

handyman

WHY THIS MATTERS

"Most agent failures aren't model problems — they're interface problems. If your tool call surface is underspecified, the agent can't plan, can't validate, and can't recover. A well-designed tool schema gives the model a deterministic affordance: it knows what it can ask for, what it will get back, and how to react when something goes wrong."

1. Use discriminated unions for results

Return a single top-level shape that is always valid JSON and always parseable. The agent can branch on ok without brittle string matching. 

// shared/types/tool-result.ts
export type ToolOk<T> = {
  ok: true;
  data: T;
  meta?: { requestId?: string; durationMs?: number };
};

export type ToolErr = {
  ok: false;
  error: {
    code:
      | "INVALID_INPUT"
      | "NOT_FOUND"
      | "UNAUTHORIZED"
      | "RATE_LIMITED"
      | "CONFLICT"
      | "UPSTREAM_FAILURE"
      | "TIMEOUT"
      | "INTERNAL";
    message: string;
    retryable: boolean;
    details?: unknown;
  };
  meta?: { requestId?: string; durationMs?: number };
};

export type ToolResult<T> = ToolOk<T> | ToolErr;

2. Validate inputs with Zod and echo normalized args

Agents often send nearly-correct inputs. Normalize them and return what you actually used — when a later step uses the tool output as context, the agent sees the exact canonical args. 

import { z } from "zod";

export const SearchTicketsInput = z.object({
  query: z.string().min(1),
  limit: z.number().int().min(1).max(50).default(20),
  cursor: z.string().optional(),
});

export type SearchTicketsArgs = z.infer<typeof SearchTicketsInput>;

export type SearchTicketsData = {
  normalized: SearchTicketsArgs; // ← echo back what was used
  items: Array<{ id: string; title: string; status: "open" | "closed" }>;
  nextCursor?: string;
};

3. Design error codes for agent decisions

Humans like messages; agents need categories. Good error codes are actionable (imply a next step), stable (don't change across refactors), and complete (cover expected failures).

Error Code Agent Action
INVALID_INPUTFix args (no retry)
RATE_LIMITEDRetry with backoff
UPSTREAM_FAILURESwitch provider / degrade gracefully
CONFLICTRe-read state then retry
NOT_FOUNDAbort or ask user to clarify

4. Make retries safe with idempotency keys

Agents are inherently retry-prone — timeouts, tool glitches, partial context. Concurrency guards prevent double writes. 

export const CreateInvoiceInput = z.object({
  idempotencyKey: z.string().min(10), // ← agent must supply this
  customerId: z.string(),
  amountCents: z.number().int().positive(),
});

// For updates — optimistic concurrency
export const UpdateDocInput = z.object({
  docId: z.string(),
  ifVersion: z.number().int().nonnegative(), // reject stale writes
  patch: z.array(z.object({
    op: z.enum(["replace"]),
    path: z.string(),
    value: z.any()
  })),
});

5. Add a lightweight "capabilities" tool

Agents do better when they can ask what the tool can do before calling it. Reduces prompt bloat and helps the agent select the right operation without guessing. 

export type Capabilities = {
  name: string;
  version: string;
  operations: Array<{
    op: string;
    description: string;
    inputSchemaRef: string;
    outputSchemaRef: string;
    commonErrors: string[];
  }>;
};

Treat tools as public APIs: versioned, documented, and enforced with tests.

Pro Move

Bake a tool-contract test suite into CI

Add contract tests that verify every tool returns ToolResult<T>, Zod validation maps to INVALID_INPUT, retryable errors set retryable: true, and outputs remain backward compatible. 

import { describe, expect, it } from "vitest";
import { searchTickets } from "../tools/searchTickets";

describe("tool contract: searchTickets", () => {
  it("always returns ToolResult shape", async () => {
    const res = await searchTickets({ query: "" } as any);
    expect(res).toHaveProperty("ok");
    if (res.ok === false) {
      expect(res.error).toHaveProperty("code");
      expect(typeof res.error.retryable).toBe("boolean");
    }
  });
});

If you treat tool schemas as public APIs and enforce them with tests, agents become predictably capable — and failures become recoverable rather than catastrophic.
