DEVTIP #004
Tools aren't “APIs”; they're contracts: design tool schemas like you design libraries
If a tool can't fail predictably, your agent can't recover.
Agent Tooling
WHY THIS MATTERS
Most agent failures aren't “model problems” — they're interface problems. If your tool call surface is underspecified (ambiguous inputs, untyped outputs, inconsistent errors), the agent can't plan, can't validate, and can't recover. A well-designed tool schema gives the model a deterministic affordance: it knows what it can ask for, what it will get back, and how to react when something goes wrong.
1. Use discriminated unions for results
Return a single top-level shape that is always valid JSON and always parseable. The agent can branch on ok without brittle string matching.
 
// shared/types/tool-result.ts
export type ToolOk<T> = {
  ok: true;
  data: T;
  meta?: { requestId?: string; durationMs?: number };
};

export type ToolErr = {
  ok: false;
  error: {
    code:
      | "INVALID_INPUT"
      | "NOT_FOUND"
      | "UNAUTHORIZED"
      | "RATE_LIMITED"
      | "CONFLICT"
      | "UPSTREAM_FAILURE"
      | "TIMEOUT"
      | "INTERNAL";
    message: string;
    retryable: boolean;
    details?: unknown;
  };
};

export type ToolResult<T> = ToolOk<T> | ToolErr;
2. Validate inputs with Zod and echo normalized args
Agents often send nearly-correct inputs. Normalize them and return what you actually used — when a later step uses the tool output as context, the agent sees the exact canonical args.
 
export const SearchTicketsInput = z.object({
  query: z.string().min(1),
  limit: z.number().int().min(1).max(50).default(20),
  cursor: z.string().optional(),
});

export type SearchTicketsData = {
  normalized: SearchTicketsArgs; // echo back what was used
  items: Array<{ id: string; title: string; status: "open" | "closed" }>;
  nextCursor?: string;
};
3. Design error codes for agent decisions
Humans like messages; agents need categories. Good codes are actionable, stable, and complete:
• INVALID_INPUT → fix args (no retry)
• RATE_LIMITED → retry with backoff
• UPSTREAM_FAILURE → switch provider / degrade
• CONFLICT → re-read state then retry
4. Make retries safe with idempotency keys
Agents are inherently retry-prone. Idempotency keys and optimistic concurrency prevent double-writes.
 
export const CreateInvoiceInput = z.object({
  idempotencyKey: z.string().min(10), // agent must supply this
  customerId: z.string(),
  amountCents: z.number().int().positive(),
});

// For updates - optimistic concurrency
export const UpdateDocInput = z.object({
  docId: z.string(),
  ifVersion: z.number().int().nonnegative(), // reject stale writes
  patch: z.array(z.object({
    op: z.enum(["replace"]),
    path: z.string(),
    value: z.any()
  })),
});
5. Add a lightweight "capabilities" tool
Agents do better when they can discover what a tool can do before calling it. Reduces prompt bloat and wrong-operation calls.
 
export type Capabilities = {
  name: string;
  version: string;
  operations: Array<{
    op: string;
    description: string;
    inputSchemaRef: string;
    outputSchemaRef: string;
    commonErrors: string[];
  }>;
};
PRO MOVE
Bake a tool-contract test suite into CI
Add contract tests verifying every tool returns ToolResult<T>, Zod validation maps to INVALID_INPUT, retryable errors set retryable: true, and outputs are backward compatible.
describe("tool contract: searchTickets", () => {
  it("always returns ToolResult shape", async () => {
    const res = await searchTickets({ query: "" } as any);
    expect(res).toHaveProperty("ok");
    if (res.ok === false) {
      expect(res.error).toHaveProperty("code");
      expect(typeof res.error.retryable).toBe("boolean");
    }
  });
});
