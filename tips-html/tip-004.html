DEVTIP #004
Tools arent APIs; theyre contracts: design tool schemas like you design libraries
If a tool cant fail predictably, your agent cant recover.
WHY THIS MATTERS
Most agent failures arent model problems  theyre interface problems. If your tool call surface is underspecified (ambiguous inputs, untyped outputs, inconsistent errors), the agent cant plan, cant validate, and cant recover. A well-designed tool schema gives the model a deterministic affordance: it knows what it can ask for, what it will get back, and how to react when something goes wrong.
The core idea: make tools machine-legible and failure-aware
Below are a few concrete patterns that make tool calls dramatically more reliable in real agents (especially multi-step, retry-prone workflows).
1) Use discriminated unions for results
Return a single top-level shape that is always valid JSON and always parseable.
// shared/types/tool-result.ts\nexport type ToolOk<T> = {\n ok: true;\n data: T;\n meta?: {\n requestId?: string;\n durationMs?: number;\n };\n};\n\nexport type ToolErr = {\n ok: false;\n error: {\n code:\n | \"INVALID_INPUT\"\n | \"NOT_FOUND\"\n | \"UNAUTHORIZED\"\n | \"RATE_LIMITED\"\n | \"CONFLICT\"\n | \"UPSTREAM_FAILURE\"\n | \"TIMEOUT\"\n | \"INTERNAL\";\n message: string;\n retryable: boolean;\n details?: unknown;\n };\n meta?: {\n requestId?: string;\n durationMs?: number;\n };\n};\n\nexport type ToolResult<T> = ToolOk<T> | ToolErr;
Why: the agent can branch immediately on ok without brittle string matching.
2) Validate inputs with Zod and echo normalized inputs
Agents often send nearly-correct inputs. Normalize them and return what you actually used.
import { z } from \"zod\";\n\nexport const SearchTicketsInput = z.object({\n query: z.string().min(1),\n limit: z.number().int().min(1).max(50).default(20),\n cursor: z.string().optional(),\n});\n\nexport type SearchTicketsArgs = z.infer<typeof SearchTicketsInput>;
 
export type SearchTicketsData = {\n normalized: SearchTicketsArgs;\n items: Array<{ id: string; title: string; status: \"open\" | \"closed\" }>;\n nextCursor?: string;\n};
Why: when a later step uses the tool output as context, the agent sees the exact canonical args.
3) Design error codes for agent decisions
Humans like messages; agents need categories. Good error codes are actionable, stable, and complete. Example mapping:
INVALID_INPUT Fix args (no retry)
RATE_LIMITED Retry with backoff
UPSTREAM_FAILURE Switch provider / degrade gracefully
CONFLICT Re-read state then retry
4) Make retries safe: idempotency + optimistic concurrency
If the agent retries, you must guarantee the operation is safe.
export const CreateInvoiceInput = z.object({\n idempotencyKey: z.string().min(10),\n customerId: z.string(),\n amountCents: z.number().int().positive(),\n});
 
export const UpdateDocInput = z.object({\n docId: z.string(),\n ifVersion: z.number().int().nonnegative(),\n patch: z.array(z.object({ op: z.enum([\"replace\"]), path: z.string(), value: z.any() })),\n});
Why: agents are inherently retry-prone; concurrency guards prevent double writes.
5) Add a lightweight capabilities tool
Agents do better when they can ask what the tool can do.
export type Capabilities = {\n name: string;\n version: string;\n operations: Array<{\n op: string;\n description: string;\n inputSchemaRef: string;\n outputSchemaRef: string;\n commonErrors: string[];\n }>;\n};
PRO MOVE
Bake a tool-contract test suite into CI. Add contract tests that verify: every tool returns the ToolResult<T> shape, Zod validation maps to INVALID_INPUT, retryable errors set retryable: true, and outputs remain backward compatible (snapshot the schema). Agents become predictably capable  and failures become recoverable rather than catastrophic.
 
import { describe, expect, it } from \"vitest\";\nimport { searchTickets } from \"../tools/searchTickets\";\n\ndescribe(\"tool contract: searchTickets\", () => {\n it(\"always returns ToolResult shape\", async () => {\n const res = await searchTickets({ query: \"\" } as any);\n expect(res).toHaveProperty(\"ok\");\n if (res.ok === false) {\n expect(res.error).toHaveProperty(\"code\");\n expect(typeof res.error.retryable).toBe(\"boolean\");\n }\n });\n});
