DEVTIP #001

Context Windows as Architecture

Feed Agents the Right Slice

WHY THIS MATTERS

An LLM agent is only as good as the context you hand it — structuring your SvelteKit codebase so that any feature's full picture fits in a single, coherent context window is the highest-leverage architectural decision you can make when working with coding agents.

The Problem with Unstructured Codebases

Most LLM agents fail not because of model quality, but because the developer pastes in the wrong files — or too many of them. A 200k-token context window is still a finite, attention-diluted resource. Dumping your entire src/ directory guarantees the agent hallucinates missing imports, invents non-existent stores, or contradicts your actual data layer.

The fix: make your codebase legible to an agent by design.

File Structure as a Signal Layer

Organize by feature, not by type. Instead of:

src/
  components/
    Button.svelte
    UserCard.svelte
    InvoiceTable.svelte
  stores/
    userStore.ts
    invoiceStore.ts
  api/
    users.ts
    invoices.ts

Use a co-located feature structure:

src/
  features/
    users/
      UserCard.svelte
      userStore.ts
      userService.ts
      user.types.ts
      index.ts          ← barrel export
    invoices/
      InvoiceTable.svelte
      invoiceStore.ts
      invoiceService.ts
      invoice.types.ts
      index.ts
  routes/
    users/
      +page.svelte
      +page.server.ts
    invoices/
      +page.svelte
      +page.server.ts

When you ask an agent to "add a status badge to the invoice feature," you hand it:

features/invoices/

That's it. Every relevant type, store, component, and service is in one folder. No cross-folder hunting. No context pollution from unrelated features.

Barrel Exports as Agent-Readable Manifests

Your index.ts barrel is not just a convenience export — it's a declaration of the feature's public surface. Write it intentionally:

// features/invoices/index.ts

// Types
export type { Invoice, InvoiceStatus, InvoiceLineItem } from './invoice.types';

// Service (server-side logic)
export { getInvoices, createInvoice, updateInvoiceStatus } from './invoiceService';

// Store (client-side state)
export { invoiceStore, selectedInvoice } from './invoiceStore';

// Components
export { default as InvoiceTable } from './InvoiceTable.svelte';
export { default as InvoiceStatusBadge } from './InvoiceStatusBadge.svelte';

When an agent reads this file first, it instantly maps the entire feature's capabilities without opening a single implementation file. Prompt pattern: "Read features/invoices/index.ts to understand what's available, then implement X."

Co-locate Types — Always

Types defined far from their usage force the agent to chase references. Put invoice.types.ts in the feature folder and import from it everywhere:

// features/invoices/invoice.types.ts

export type InvoiceStatus = 'draft' | 'sent' | 'paid' | 'overdue';

export interface InvoiceLineItem {
  id: string;
  description: string;
  quantity: number;
  unitPrice: number; // in cents
}

export interface Invoice {
  id: string;
  clientId: string;
  status: InvoiceStatus;
  lineItems: InvoiceLineItem[];
  createdAt: Date;
  dueAt: Date;
  totalCents: number;
}

When the agent encounters Invoice anywhere in context, the full shape is right there — no need to resolve it across the project graph.

Slicing Context Per Task

Build a personal discipline of context budgeting. For each agent task, ask:

  • Feature context: the feature folder (features/invoices/)
  • Route context: the relevant +page.server.ts or +server.ts
  • Design system context: your token definitions or a DESIGN_SYSTEM.md (see Tip #3)
  • Constraint file: a AGENTS.md at the project root with your coding conventions

Avoid including: unrelated features, your entire lib/ directory, node_modules (obviously), or boilerplate config files.

A tight 4-file context will outperform a 40-file dump every time.

PRO MOVE

AGENTS.md as a Living Contract

Create a AGENTS.md at your project root and include it in every agent session. Keep it under 200 lines:

# Agent Context: MyApp

## Stack
- SvelteKit 2, TypeScript strict mode, Tailwind v4
- PostgreSQL via Drizzle ORM, DuckDB for analytics, Pocketbase for auth
- Zod for all I/O validation

## Conventions
- All DB calls go through `src/lib/db/` service modules — never query directly in routes
- Use `$lib` alias for all internal imports
- Components are never async — async logic lives in `+page.server.ts` or service modules
- All monetary values stored in cents (integer), displayed via `formatCurrency()` util

## File patterns
- Feature types: `features/[name]/[name].types.ts`
- Feature barrel: `features/[name]/index.ts`
- API routes: `src/routes/api/[resource]/+server.ts`

This file makes every agent session context-aware from line one, with zero repetition on your end.
