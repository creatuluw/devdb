DEVTIP #009
RAG Quality Gates for Production Agents
Measure retrieval before the model sees it
Knowledge Bases
WHY THIS MATTERS
If retrieval is noisy, your agent will confidently answer from the wrong chunks. A simple quality gate in front of generation turns RAG from a demo into a dependable system: fail fast, ask for clarification, or broaden retrieval before you burn tokens on an answer.
1. Define a retrieval contract (what good looks like)
Treat retrieval as an API with a typed output. The agent should never see raw vector hits; it should see a normalized set of candidates with scores and metadata you can test and log.
// src/lib/rag/types.ts
export type DocChunk = {
  id: string;
  docId: string;
  title: string;
  url: string;
  text: string;
  score: number;
};

export type RetrievalResult = {
  query: string;
  candidates: DocChunk[];
  topScore: number;
  uniqueDocs: number;
};
2. Add a quality gate before generation
Gate on a few cheap signals: top score, number of unique docs, and how concentrated the evidence is. If the gate fails, return an explicit tool error so the agent can ask a follow-up question or broaden the query.
// src/lib/rag/gate.ts
import type { RetrievalResult } from "$lib/rag/types";

export type GateDecision =
  | { ok: true }
  | { ok: false; reason: string; hint: string };

export function gateRetrieval(r: RetrievalResult): GateDecision {
  if (r.candidates.length === 0)
    return { ok: false, reason: "NO_HITS", hint: "Try broader keywords" };

  if (r.topScore < 0.72)
    return { ok: false, reason: "LOW_CONFIDENCE", hint: "Ask a clarifying question" };

  if (r.uniqueDocs < 2)
    return { ok: false, reason: "TOO_NARROW", hint: "Expand retrieval to more documents" };

  return { ok: true };
}
3. Wire it into a SvelteKit server tool
Expose retrieval as a server-side tool that returns either a clean context pack or a structured failure. This lets your agent planner decide to re-query, ask the user, or proceed with generation.
// src/routes/api/tools/retrieve/+server.ts
import { json } from "@sveltejs/kit";
import { gateRetrieval } from "$lib/rag/gate";
import { searchVector } from "$lib/rag/vector";

export async function POST({ request }) {
  const { query } = await request.json();
  const r = await searchVector(query);
  const decision = gateRetrieval(r);

  if (!decision.ok) {
    return json({
      ok: false,
      error: { code: decision.reason, message: decision.hint, retryable: true }
    });
  }

  const context = r.candidates.slice(0, 6).map((c) => ({
    title: c.title,
    url: c.url,
    text: c.text
  }));

  return json({ ok: true, data: { query, context } });
}
4. Add a retrieval eval test you can run in CI
Create a tiny golden set of queries with expected documents. Your test does not need perfect recall; it needs to catch regressions when chunking, embeddings, or filters change.
// tests/rag/golden.test.ts
import { describe, it, expect } from "vitest";
import { searchVector } from "$lib/rag/vector";

const GOLDEN = [
  { query: "reset password", mustContainDocId: "auth-playbook" },
  { query: "webhook retries", mustContainDocId: "webhooks" },
];

describe("rag: golden retrieval", () => {
  it.each(GOLDEN)("retrieves %s", async ({ query, mustContainDocId }) => {
    const r = await searchVector(query);
    const docIds = new Set(r.candidates.map((c) => c.docId));
    expect(docIds.has(mustContainDocId)).toBe(true);
  });
});
PRO MOVE
Log a compact retrieval trace for every agent run: query, topScore, uniqueDocs, docIds, and gate outcome. Then add a weekly job that samples low-confidence runs and turns them into new golden queries. This is the fastest path to a knowledge base that improves with real usage, not guesswork.
